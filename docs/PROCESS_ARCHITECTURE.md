# Timer Execution Flow

Timer execution pipeline:

UI Start
↓
useTimerEngine.start()
↓
TimerScheduler.start()
↓
TimerGraph constructed from TimerNodeConfig
↓
activateNode(root)
↓
activateNode(parallel siblings)
↓
Timer countdown loop begins
↓
update remaining time map
↓
node reaches 0
↓
play alarm sound
↓
completeNode(node)
↓
activate sequential child (if exists)
↓
parallel nodes continue independently
↓
scheduler stops when no active nodes remain

Notes:

• Parallel nodes activate **when the parent activates**, not when it finishes.  
• Sequential children activate **after parent completion**.  
• Multiple nodes may run simultaneously.

# Audio Playback Flow

TimerScheduler
↓
resolveSound(node)
↓
AudioManager.play(sound)
↓
HTMLAudioElement created
↓
registerActive(audio)
↓
sound plays independently

Characteristics:

• Multiple alarms may overlap  
• AudioManager tracks active sounds  
• stopAll() stops all currently playing alarms

# UI Rendering Flow

Application UI rendering pipeline:

TimerManager
↓
TimerRunner
↓
useTimerEngine hook
↓
TimerScheduler runtime state
↓
remaining time Map<string, number>
↓
Runner Skin (ColumnRunnerSkin)
↓
TimerColumnGrid layout
↓
TimerNodeView components

Rendering is **skin controlled**.

Important behavior:

• Timers render **before start**  
• If remaining time does not exist yet → fallback to `node.durationMs`  
• Parallel timers appear as **columns**  
• Sequential timers appear as **rows**

# Editor Layout Rules

Editor uses a **column-aligned timer tree layout**.

### Example timer structure

Main  
├ Parallel A  
├ Parallel B  
│ └ Child B1  
└ Child A

### Grid layout representation

| Main | Parallel A | Parallel B |
|------|------------|------------|
|ChildA|            |Child B1    |

### Layout rules

Parent → first column

Parallel nodes  
→ appear as additional columns to the right

Sequential children  
→ appear as rows below their parent column

Layout is generated by:

TimerColumnGrid

Used by:

• ColumnEditorSkin  
• ColumnRunnerSkin

# Skin Architecture

Skins control **visual layout and appearance**.

The runtime system separates:

• execution (TimerScheduler)  
• rendering (skins)

---

### Skin Interface

TimerSkin

Required method:

renderRunner(root, remaining)

Where:

root → TimerNodeConfig tree  
remaining → Map of nodeId → remaining milliseconds

---

### Skin Rendering Pipeline

TimerRunner
↓
TimerSkin.renderRunner()
↓
TimerColumnGrid
↓
renderNode(node)
↓
TimerNodeView

---

### Skin Resolution

Skins are **resolved per node** using:

TimerViewModel.resolveSkin()

Resolution order:

node.skin  
↓  
parent skin (if inheritSkin = true)  
↓  
default skin

This allows:

• different skins for parallel alarms  
• inheritance for nested nodes

# Persistence Model

Timers are stored as:

TimerNodeConfig tree structures.

Storage backend:

localStorage via PersistenceService.

---

### Persisted data

Saved timers include:

• timer tree structure  
• node durations  
• node sounds  
• node skins  
• parallel / sequential relationships

Important change:

Skins are now stored **per node**, not globally.

# ViewModel Layer

TimerViewModel.ts provides **view-layer helpers**.

Current responsibilities:

resolveSkin(node, parentSkin)

Purpose:

• isolate UI logic from engine logic  
• allow skins to remain simple  
• centralize rendering rules

Future responsibilities may include:

• active node highlighting  
• runtime UI formatting
